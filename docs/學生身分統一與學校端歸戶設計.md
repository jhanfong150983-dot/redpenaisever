# 學生身分統一與學校端歸戶設計（SSoT）

> 狀態：**設計定稿、尚未實作**（2026-06-03）。schema 為設計稿，未 apply 到 production。
> 目的：學校端 toB MVP 的承重牆——把分散在各老師手上的學生資料，歸戶成「一個學生 / 一所學校」，才能做跨科學習檔案與弱點地圖。

## 0. 背景與現況

- 今天多租戶全靠 `owner_id`（老師 uuid）。`schools` / `memberships` 表尚未落地。
- `students` 欄位：id, classroom_id, seat_number, name, owner_id, email, **auth_user_id**, **provider_student_id**(1campus studentID), student_number(校方學號)。
- 既有匯入管線：
  - 1campus 同步（`api/auth/1campus.js`、`api/data/[action].js`）：每位老師同步自己的 1campus 班，建 `classrooms`(owner_id, folder='1Campus')、`upsert_students_batch` 寫學生（去重鍵 = owner_id+classroom_id+seat_number），再補 email/provider_student_id/student_number。
  - 學生 1campus SSO 登入：依 學號→email→student_number→班座號 找 student 列、綁 auth_user_id（可跨多 owner）。
  - `campus_classroom_sync` 表含 **`provider_dsns`** = 1campus 學校代碼（學校身分其實已隱含存在）。
- 2026-06-03 唯讀盤點（production）：3042 筆 students → 可歸 2444 筆 → **1257 真人**、266 人跨老師、**0 錯併**、598 真孤兒（多為測試班/整班未接名冊）。三個錨（學號/帳號/email）在現有資料皆 0 姓名衝突。

**核心問題**：去重鍵是「老師+班+座號」而非學號，所以同一人在多位老師的班 = 多筆獨立 students 列，沒有任何東西把它們連成一個人。

## 1. 設計原則

1. **Additive、可逆**：只新增 3 張表 + 2 個 nullable 欄位。`students` / `submissions` 一筆都不改，drop 新表即還原。
2. **身分與在籍分離**：`school_person` 只放「一輩子不變的」；年級/班級/座號是「每學年的在籍紀錄」，放在 students 列 + `classrooms.school_year`。→ 升年級/換班不需調整身分。
3. **歸戶錨優先序**：學號（1campus studentID / 校方學號）> email > 人工（班+座號+姓名確認）。
4. **永不自動按姓名合併**（盤點證實姓名在錨內 0 衝突，但跨錨的純姓名比對仍須人工）。
5. **跨校不歸戶**：person 以「學校」為範圍，轉學到新校 = 新 person（符合隱私邊界）。

## 2. Schema 設計稿（尚未 apply）

```sql
-- 1) 學校：1campus 以 dsns 為錨；非 1campus 為 'manual'
create table schools (
  id            text primary key,                 -- 'sch_...'
  name          text not null,
  provider      text not null default 'manual',   -- 'campus1' | 'manual'
  provider_dsns text,                             -- 1campus 學校代碼（manual 為 null）
  created_at    timestamptz default now(),
  updated_at    timestamptz default now()
);
create unique index schools_dsns_uniq
  on schools(provider_dsns) where provider_dsns is not null;

-- 2) 校級「人」= 歸戶 SSoT（同校同學號 = 同一人）。只放永久不變的欄位。
create table school_person (
  id                  text primary key,           -- 'sp_...'
  school_id           text not null references schools(id),
  name                text not null,
  student_number      text,                       -- 校方學號（人看的）
  provider_student_id text,                       -- 1campus studentID（最穩的錨）
  email               text,
  status              text not null default 'active', -- 'active'|'graduated'|'transferred'
  created_at          timestamptz default now(),
  updated_at          timestamptz default now()
);
create unique index sp_school_provider_uniq
  on school_person(school_id, provider_student_id) where provider_student_id is not null;
create unique index sp_school_studentno_uniq
  on school_person(school_id, student_number) where student_number is not null;
create index sp_school_idx on school_person(school_id);

-- 3) 對應表：per-classroom 的 students 列 → 一個 person（students 完全不動）
create table student_person_link (
  student_id   text primary key references students(id), -- 一筆學生列只對一個人
  person_id    text not null references school_person(id),
  match_source text not null,    -- 'provider_student_id'|'email'|'student_number'|'manual'
  confidence   text not null default 'auto', -- 'auto' | 'manual_confirmed'
  created_at   timestamptz default now()
);
create index spl_person_idx on student_person_link(person_id);

-- 4) 既有表唯一的非破壞變更（皆 nullable）：
alter table classrooms add column school_id   text references schools(id);
alter table classrooms add column school_year text;  -- 學年度，例 '113'。升年級的關鍵欄位。
```

跨科 / 跨年聚合查詢：`submissions → students → student_person_link → school_person`，
「目前班級」= 該 person 最新 `classrooms.school_year` 的 students 列。`submissions.student_id` 永遠不動。

## 3. 升年級 / 換班的處理（國小換班情境）

身分永久、在籍年度化，所以**不需要任何遷移動作**：

```
王小明  學號 P
 ├─ 113學年 601班 5號  → students 列 A ─┐
 └─ 114學年 701班 12號 → students 列 B ─┴→ 同一個 school_person (school_id + P)
```

每學年 1campus 同步只是多一筆 students 列（新 classroom，帶 school_year），靠永久學號自動掛回同一個 person。person 自然累積跨年歷史（= 機構記憶）。
- 留級/跳級：自然涵蓋（只是該年的在籍紀錄）。
- 轉學出/入：到新校為新 person。
- 學號於同校期間變更（罕見）：會斷錨→產生新 person；緩解=email/auth 次級比對 + 未來人工 merge 工具。

## 4. 入口 A：1campus（自動，主路）

在現有同步尾端（`[action].js` upsert 之後）多一步：
1. 依 `provider_dsns` upsert `schools`（name 取 1campus 校名，缺則先用 dsns、可改）；補該班 `classrooms.school_id` 與 `school_year`。
2. 每個學生：依 (school_id, **provider_student_id**) upsert `school_person`，刷新 name/student_number/email。
3. upsert `student_person_link`（student.id → person, source='provider_student_id'）。

→ 同學號跨多位老師自動歸成一個人，主任與各科老師共用同一份名冊，學校零額外輸入。

## 5. 入口 B：非 1campus（Excel 名冊，備援）

學校無 1campus 時，主任上傳名冊檔：

| 班級 | 座號 | 姓名 | 學號 | email(選填) |
|---|---|---|---|---|

1. 建 `schools`（provider='manual'，自生 id）。
2. 逐列：依班級名 ensure `classrooms`（掛 school_id + school_year）、ensure `students`（沿用 `upsert_students_batch`，鍵=班+座號）。
3. 依 (school_id, **學號**) upsert `school_person`、寫 `student_person_link`（source='student_number'）。
4. 無學號的列 → 退 email；再無 → `confidence='manual'` 進人工確認清單。

→ 與 1campus 殊途同歸，最後都落到同一個 `school_person`，差別只在錨。

## 6. 欄位取捨

- **現在加**：`classrooms.school_year`（升年級必要）、`school_person.status`（畢業/轉出排除）。
- **刻意不加（YAGNI/隱私）**：家長/監護人（家長端再開獨立表）、生日（敏感、用不到）、性別（報表要再補）、年級/班級放 person（會被升年級打爆，絕不）。

## 7. 既有資料（legacy）回填政策

- 2026-06-03 決定：**暫緩回填**既有 3042 筆（多為測試資料、老師自用、測試階段）。
- 未來計畫：強制老師輸入學生 email 當錨後再回填。
- 本設計**只往前對試辦校生效**（forward-only），不動既有資料。
